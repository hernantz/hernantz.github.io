Reflections on Local first apps

Published on 2022-06-04, under Programming.

While architecting a non-cloud app (like the good ol' desktop apps or mobile apps), you need to think what are the use cases for an always-on server in the cloud to act as a support service for the app.

  1. Share state and updates between users, since these users might not be connected at the same time.
  2. Share state between same user devices or different install/uninstall cycles. This is a special case of the previous item.
  3. Ensure deliverability or execution. For example, turn on an alarm, send a scheduled alert, etc. This actions should happen, no matter if the app is closed, has been uninstalled, or the user doesn't have access to the internet for example.
  4. Store massive amounts of data. The non-cloud app it's just a client that consumes data from a tiny window without storing all the data locally, but synchronize every now and then. Like email clients, or gallery/photo apps.

The way this synchronization happens will depend on what type of data needs to be shared. If it's some sort of collaboration framework, then you need primitives for eventual consistency (CRDTs), sometimes, if each piece of data is immutable then a shared drive/filesystem could suffice (like a photo gallery app) it just reads and writes files. Sometimes these updates happen in realtime or in batches (git) and some form of conflict resolution mechanism is needed.

Another interesting pattern is to mount remote filesystems, plan9 is an exotic OS that had this idea baked in, where you could share resources from other computers by mounting filesystems over the network.
